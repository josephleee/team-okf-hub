OKF Hub M1–M3 shipped

팀 지식 허브 & AI 에이전트 org 메모리 — 사용 가이드

OKF Hub는 팀의 OKF 번들(Markdown+YAML 파일의 git 저장소)을 브라우즈·검색·그래프로 다룰 수 있는 지식 허브이자, 에이전트가 완료한 작업을 기록하고 다시 조회하는 조직 메모리입니다. 같은 저장소를 사람은 웹으로, AI 에이전트는 MCP·REST로 읽고 씁니다.

1 OKF Hub가 무엇인가

OKF(Open Knowledge Format)는 조직의 지식(테이블 스키마, 지표 정의, 런북 등)을 Markdown + YAML frontmatter 파일로 표현하는 Google Cloud의 개방 규격입니다. 각 파일 하나가 하나의 concept이고, 필수 필드는 type 하나뿐이며, 파일끼리 일반 Markdown 링크로 연결되어 그래프를 이룹니다.

OKF Hub는 이 "그냥 파일" 번들 위에 두 가지 레이어를 얹습니다:

📚 팀 지식 허브

사람을 위한 웹 UI. 개념을 브라우즈·검색하고, 개념 그래프를 탐색하고, 브라우저에서 바로 편집(실시간 검증 + 새니타이즈 미리보기)합니다.

🧠 에이전트 org 메모리

AI 에이전트를 위한 MCP·REST 인터페이스. 에이전트가 완료한 작업을 WorkRecord로 기록(write)하고, 같은 메모리를 다시 조회(read)합니다.

핵심 원칙 — git이 source of truth. OKF Hub가 번들 외에 저장하는 것은 언제든 git에서 다시 만들 수 있는 일회용 SQLite 인덱스뿐입니다. 지식이 특정 DB에 갇히지 않습니다.

2 빠른 시작 (2분)

1

설치

git clone https://github.com/josephleee/team-okf-hub.git
cd team-okf-hub
npm install
2

실행

기본 예제 번들(bundles/example)로 바로 띄우기:

npm run dev            # http://localhost:3000

여러분의 번들과 에이전트 쓰기 토큰을 지정하려면:

export OKF_BUNDLE_DIR="/path/to/your/okf-bundle"   # 기본값: bundles/example
export OKF_INGEST_TOKEN="$(openssl rand -hex 24)"  # 쓰기(인제스션)에 필요
npm run dev
3

열기

localhost:3000 — 홈(타입별 개념 + 그래프) · /search?q= · /work(작업 기록 타임라인) · 개념 페이지의 edit / 상단 + new.

환경변수용도기본값
OKF_BUNDLE_DIR서비스할 OKF 번들 디렉터리bundles/example
OKF_INGEST_TOKEN쓰기(인제스션) Bearer 토큰. 미설정 시 모든 쓰기는 503으로 거부(없음)

3 지식 허브로 쓰기 (사람)

페이지설명
/타입별 개념 목록 + 인터랙티브 개념 그래프(홈)
/concept/<path>렌더된 개념 + 메타데이터 + 아웃바운드 링크 + 백링크
/search?q=전문 검색(SQLite FTS5), 스니펫 하이라이트
/graph전체 개념 그래프 탐색기
/workWorkRecord 타임라인(날짜별, actor/project 필터)
/concept/new, /edit/<path>개념 생성 / 편집 에디터

브라우저에서 편집하기 ✅ M2a

개념을 열고 edit, 또는 상단 + new로 생성합니다. 에디터는 입력하는 동안 okf-core로 실시간 검증하고(필수 필드는 type 하나), 새니타이즈된 미리보기를 보여줍니다. Save하면 번들 디렉터리의 .md 파일에 쓰고 인덱스를 다시 만듭니다. 영구 보존은 git 커밋으로:

cd "$OKF_BUNDLE_DIR" && git add -A && git commit -m "edit via OKF Hub"
참고: 현재는 로컬 번들 파일에 씁니다. GitHub 로그인 + 자동 PR 생성은 다음 마일스톤(M2b)입니다.

4 org 메모리로 쓰기 (에이전트)

핵심 아이디어: 에이전트가 완료한 작업을 WorkRecord로 남기고, 같은 메모리를 다시 조회합니다. "무엇을 했는가 / 누가 무엇을 했는가"가 조직 전체가 공유하는 검증된 활동 로그가 됩니다.

✍️ 기록 (write)

MCP 툴 okf_record_work 또는 POST /api/v1/work. 토큰 필요. 입력은 검증·새니타이즈되어 work/…​.md 파일로 저장됩니다.

🔎 조회 (read)

okf_recent_work · okf_search · okf_get · okf_graph, 또는 GET /api/v1/*. MVP에서 REST 읽기는 열림(MCP 엔드포인트는 읽기도 토큰 필요).

두 가지 쓰기 경로를 지식의 종류에 맞춰 구분합니다:

지식 종류쓰기 방법왜
정식 지식(canon)
테이블·지표·런북 정의		사람의 edit → 검증 → (PR) 흐름. 에이전트는 직접 못 씀."정식 진실"은 리뷰를 거쳐야 신뢰됨
작업 기록(WorkRecord)
"무엇을 했다"		에이전트가 okf_record_work로 직접 emit(토큰·검증·새니타이즈)추가 전용 활동 로그 → 가벼운 경로가 적절

5 Claude Code에 연결하기 (MCP)

가장 강력한 사용법입니다. OKF Hub를 MCP 서버로 등록하면 모든 Claude 세션에서 아래 5개 툴을 바로 쓸 수 있습니다 — 세션이 org 메모리를 읽고, 완료한 작업을 다시 기록합니다.

1

서버를 토큰과 함께 실행

OKF_INGEST_TOKEN="$(openssl rand -hex 24)" \
OKF_BUNDLE_DIR="/path/to/your/okf-bundle" \
npm run dev            # http://localhost:3000/api/mcp
2

Claude Code에 MCP 서버 등록

claude mcp add --transport http okf-hub \
  http://localhost:3000/api/mcp \
  --header "Authorization: Bearer $OKF_INGEST_TOKEN"

엔드포인트 URL은 /api/mcp입니다(내부적으로 [transport] 라우트로 처리).

3

사용

세션에서: "오늘 한 작업을 org 메모리에 기록해줘" → Claude가 okf_record_work 호출. "team-okf-hub 프로젝트 최근 작업 보여줘" → okf_recent_work 호출.

제공되는 MCP 툴

툴인자동작
okf_record_work (쓰기)title, summary, actor, project?, tags?, artifacts?, links?완료 작업을 WorkRecord로 기록
okf_recent_workproject?, actor?, limit?최근 WorkRecord 목록(최신순)
okf_searchquery메모리 전문 검색
okf_getpath개념/WorkRecord 전체 내용
okf_graphpath, depth?개념의 그래프 이웃
인증: okf_record_work가 쓰기 툴이므로 MCP 엔드포인트 전체가 토큰으로 보호됩니다. 토큰 없는 요청은 401, OKF_INGEST_TOKEN 미설정 시 503.

6 REST API (/api/v1)

MCP를 쓰지 않는 도구(CI 스크립트, curl, 다른 언어)용 미러입니다.

메서드 · 경로인증설명
POST /api/v1/workBearerWorkRecord 기록(인제스션)
GET /api/v1/work?project=&actor=&limit=open최근 WorkRecord 목록
GET /api/v1/search?q=open전문 검색
GET /api/v1/concept?path=open개념 하나의 전체 내용
GET /api/v1/graph?path=&depth=open개념의 그래프 이웃

기록 (쓰기 — 토큰 필요)

curl -X POST http://localhost:3000/api/v1/work \
  -H "Authorization: Bearer $OKF_INGEST_TOKEN" \
  -H 'content-type: application/json' \
  -d '{
    "title": "Add rehype-sanitize to render path",
    "summary": "Hardened the Markdown render path against stored XSS.",
    "actor": "jungsup",
    "project": "team-okf-hub",
    "tags": ["security","feature"],
    "artifacts": ["https://github.com/josephleee/team-okf-hub/pull/12"],
    "links": ["tables/orders.md"]
  }'
# → 201 {"ok":true,"path":"work/team-okf-hub/2026-07-01-142305-add-rehype-sanitize-to-render-path.md"}

조회 (읽기 — 열림)

curl 'http://localhost:3000/api/v1/work?project=team-okf-hub'
curl 'http://localhost:3000/api/v1/search?q=sanitize'
curl 'http://localhost:3000/api/v1/concept?path=tables/orders.md'
응답 코드: 201 기록 성공 · 401 토큰 틀림/없음 · 422 검증 실패(issues 포함) · 503 OKF_INGEST_TOKEN 미설정 · 400 잘못된 JSON 본문(POST) 또는 필수 쿼리 파라미터 누락(GET concept·graph) · 404 개념 없음.

7 WorkRecord 구조

WorkRecord는 그냥 type: WorkRecord인 OKF 개념 파일입니다. 경로는 work/<project-slug>/<YYYY-MM-DD>-<HHMMSS>-<slug>.md — project와 slug 모두 슬러그화됩니다 (project 기본값 general, slug은 title에서 생성, 시각은 timestamp에서). 예: project My Team → 디렉터리 my-team.

---
type: WorkRecord
title: Add rehype-sanitize to render path
actor: jungsup                 # 누가
project: team-okf-hub          # 어떤 프로젝트/레포
timestamp: 2026-07-01T14:23:05Z  # ISO-8601 완료 시각(없으면 서버가 채움)
tags: [security, feature]
artifacts:                     # PR/커밋/문서 링크 (복수)
  - https://github.com/josephleee/team-okf-hub/pull/12
---

Markdown 렌더 경로를 저장형 XSS로부터 강화했다.

## Related
- [orders](tables/orders.md)   ← 본문 링크는 지식 그래프의 엣지가 됨
필드필수설명
type✔ (항상 WorkRecord)OKF 필수 필드
title, summary(본문), actor✔기록 인제스션 시 필수
project—기본값 general
timestamp—ISO-8601 datetime(YYYY-MM-DDThh:mm:ss…). 날짜만 있거나(2026-07-01) 유효하지 않으면 서버 현재시각으로 대체
tags, artifacts, links—리스트. artifacts는 URL, links는 개념 경로(## Related로 추가)
안전: 에이전트 입력은 신뢰되지 않으므로 본문은 rehype-sanitize로 렌더 시 정화되고, 파일 경로는 work/ 하위 .md로만 강제됩니다(경로 탈출 차단). /work에서 아티팩트는 http(s) URL만 링크로 렌더됩니다.

8 보안 & 설정

  • 쓰기 = 토큰 게이트. POST /api/v1/work와 MCP 엔드포인트 전체가 Authorization: Bearer $OKF_INGEST_TOKEN을 요구합니다. 토큰 미설정 시 503(명시적 opt-in), 틀리면 401.
  • 입력 새니타이즈. WorkRecord 본문은 렌더 시 rehype-sanitize로 정화 — 저장형 XSS 불가.
  • 경로 봉인. 모든 쓰기는 번들 하위 .md로 제한, 적대적 timestamp/project/title도 경로를 탈출할 수 없음.
⚠️ MVP에서 REST 읽기는 열려 있습니다. GET /api/v1/*는 인증이 없어 — WorkRecord뿐 아니라 번들의 모든 개념이 서버에 접근 가능한 누구에게나 공개됩니다. (MCP 엔드포인트는 쓰기 툴을 포함하므로 읽기도 토큰이 필요합니다.) REST 읽기 인증이 추가되기 전까지 신뢰된 네트워크나 리버스 프록시 뒤에서만 운영하세요. 공개 인터넷에 그대로 노출하지 마세요.

9 CLI & 프로덕션

okf CLI (개발/CI)

npm run okf -- validate bundles/example          # 번들 검증(에러 시 exit 1)
npm run okf -- query bundles/example orders      # 전문 검색
npm run okf -- index bundles/example okf.sqlite  # SQLite 인덱스 파일 생성

프로덕션 실행

npm run build
OKF_INGEST_TOKEN=... OKF_BUNDLE_DIR=/path/to/bundle npm start   # 기본 포트 3000

테스트 / 타입체크

npm test          # Vitest (전체)
npm run typecheck # tsc --noEmit
